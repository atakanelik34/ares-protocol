# ARES Protocol Light Paper (Base Batches 003)

ARES (Autonomous Reputation & Evaluation Scoring) is Base-native trust infrastructure for autonomous AI agents. As agents begin to execute transactions, manage capital, coordinate workflows, and interact with users without constant human supervision, the ecosystem faces a core gap: there is no canonical, verifiable reputation primitive that protocols can consume as a trust signal. Most systems rely on opaque off-chain heuristics, app-specific scores, or ad hoc allowlists that do not generalize across products.

ARES addresses this by introducing a protocol-level reputation stack with cryptographic accountability and on-chain disputeability. The system is built around five components: a stake-gated canonical AgentID registry, a scorecard ledger for action-level scoring records, an ARI (Agent Reputation Index) engine, a dispute and slashing layer, and an access/integration layer for external protocols. ARES is not an app; it is an infrastructure primitive that other apps and agents can integrate.

The architecture intentionally separates ARES Core from ERC-8004 compatibility concerns. In core, AgentID is non-transferable and stake-gated, matching ARES accountability requirements. For ecosystem interoperability, ARES provides spec-accurate ERC-8004 adapters (Identity, Reputation, Validation) that expose the draft interface surface while preserving core authority boundaries. ERC-8004 was published in August 2025 and remains draft/proposed; ARES treats adapter support as compatibility infrastructure, not as a claim that the standard is finalized.

ARI is computed as a bounded 0–1000 trust score using five weighted dimensions with fixed-point math and gas-conscious decay mechanics. The scoring model keeps whitepaper parameters intact: weights [0.30, 0.25, 0.20, 0.15, 0.10], dimension scores in [0..200], time decay, and volume confidence. Instead of expensive floating-point operations, ARES uses precomputed decay factors and chunked fixed-point updates. The dispute layer can invalidate incorrect action scores and apply deterministic correction so reputation can be challenged and repaired, not only accumulated.

Economically, ARES aligns behavior through staking, slashing, and governance controls. Registration requires stake, challenge flows require stake, and accepted disputes can slash bad actors. The token utility surface includes governance and API access extension paths. For local/testnet phases, supply finalization and vesting parameters are intentionally deferred until audit and governance hardening, avoiding premature mainnet commitments while preserving credible token design direction.

Current status is testnet-validated infrastructure on Base Sepolia with live contracts, subgraph indexing, API endpoints, and dashboards. The demo scenario includes three agents, twenty action scores, and one finalized dispute with score invalidation and ARI state correction. This gives Base Batches reviewers concrete proof of execution across chain, indexing, API, and UI layers.

Why now: Base is accelerating the on-chain agent economy, but trust primitives are still fragmented. ARES provides a composable reputation substrate that can be consumed by lending, marketplaces, autonomous strategy vaults, coordination tools, and risk engines. In the next milestone window, we focus on partner integrations, parameter hardening, and path-to-mainnet with security reviews. Our objective is to make trust for agents as composable as identity and liquidity already are on Base.

We are also preparing integration templates so teams can enforce minimum trust thresholds in contracts or APIs with one call and predictable behavior.
